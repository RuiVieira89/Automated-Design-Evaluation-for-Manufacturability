

# YAML Configuration: Integration Challenges with Legacy Systems

You raise a valid concern. While YAML is excellent for human-readable, version-controlled configuration, **real-world manufacturing environments are full of legacy systems** that speak very different languages. Let's break down the friction points and practical solutions.

---

## 1. Where YAML Clashes with Legacy Systems

| Legacy System | Typical Data Format | Integration Pain |
|---|---|---|
| **ERP** (SAP, Oracle, Epicor) | SQL databases, BAPI/RFC, XML, CSV flat files | ERP systems have rigid schemas; they don't consume YAML natively |
| **MES** (Manufacturing Execution) | OPC-UA, XML, proprietary binary, SQL | Real-time systems expect structured messages, not config files |
| **CAD/CAM** (SolidWorks, NX, Mastercam) | Proprietary formats, STEP, IGES, XML-based | CAD APIs expect their own data structures; YAML is irrelevant to them |
| **PLM** (Teamcenter, Windchill) | XML, SQL, REST APIs (JSON) | PLM systems have their own data models; they won't read your YAML |
| **Quality systems** (Minitab, InfinityQS) | CSV, SQL, proprietary | Statistical tools expect tabular data |
| **Cost estimation tools** (aPriori, Costimator) | Proprietary databases, XML | Closed ecosystems with their own schemas |
| **Supplier portals / RFQ systems** | CSV, Excel, PDF, EDI (ANSI X12) | The most "legacy" of all — some still run on EDI from the 1980s |

### The Core Problem

> **YAML is a great internal representation, but it is not an interchange format that any legacy system understands natively.**

---

## 2. Specific Technical Issues

### 2.1 — Schema Rigidity vs. YAML Flexibility

YAML is schema-optional and loosely typed. Legacy systems are the opposite:

```yaml name=config/processes/cnc_milling.yaml
# YAML lets you do this — but an ERP field expects a fixed decimal
cost:
  per_part_rate_usd_per_min: 2.5   # float
  tooling_cost_usd: null            # nullable — ERP may reject this
  setup_fixed_usd: "150"            # string that looks like a number — silent bug
```

**ERP systems will reject null values, mistyped fields, or unexpected keys** without helpful error messages.

### 2.2 — Hierarchical vs. Tabular Data

YAML is naturally hierarchical. Legacy systems are overwhelmingly **tabular** (relational DB rows, CSV, flat EDI segments):

```yaml name=config/processes/injection_molding.yaml
# Natural in YAML — nested, rich
injection_molding:
  materials:
    - name: abs
      grades: [general_purpose, flame_retardant]
      cost_per_kg_usd: 2.80
    - name: nylon_pa6
      grades: [unfilled, glass_filled_30]
      cost_per_kg_usd: 4.50
```

An ERP or MES system expects this flattened:

```csv name=erp_import_materials.csv
PROCESS_CODE,MATERIAL_CODE,GRADE,COST_PER_KG_USD
INJ_MOLD,ABS,GENERAL_PURPOSE,2.80
INJ_MOLD,ABS,FLAME_RETARDANT,2.80
INJ_MOLD,PA6,UNFILLED,4.50
INJ_MOLD,PA6,GLASS_FILLED_30,4.50
```

**You will always need a flattening/mapping layer.**

### 2.3 — No Standard YAML Schema for Manufacturing

Unlike XML (which has XSD, STEP-XML, QIF, MTConnect schemas for manufacturing), YAML has **no industry-standard schemas** for:
- Process capability data
- Cost breakdowns
- Bill of materials
- Quality inspection plans

This means every integration is custom.

### 2.4 — Encoding and Character Issues

Legacy systems (especially EDI, older ERPs) often expect:
- Fixed-width or ASCII-only text
- Windows line endings (CRLF)
- Specific date formats (`YYYYMMDD`, not ISO 8601)

YAML's Unicode support and flexible formatting can produce output that older parsers choke on.

---

## 3. The Right Architecture: YAML as Internal, Adapters for External

The solution is **not to abandon YAML** — it's to treat it as your **source of truth** and build explicit adapters:

```
┌─────────────────────────────────────────────────┐
│           Your DFM System (Source of Truth)      │
│                                                  │
│   config/processes/*.yaml                        │
│   config/materials/*.yaml                        │
│   config/cost_models/*.yaml                      │
│                                                  │
│   ┌──────────────┐                               │
│   │  Pydantic     │  ← Validates & normalizes    │
│   │  Models       │    all config on load         │
│   └──────┬───────┘                               │
│          │                                       │
│   ┌──────▼───────┐                               │
│   │  Adapter      │  ← Translates to/from        │
│   │  Layer        │    external formats           │
│   └──────┬───────┘                               │
└──────────┼───────────────────────────────────────┘
           │
     ┌─────┼──────┬──────────┬──────────┬─────────┐
     ▼     ▼      ▼          ▼          ▼         ▼
   JSON   CSV    XML      SQL/DB      EDI      Excel
  (REST) (flat) (STEP,   (ERP,MES)  (supplier) (RFQ)
         files   QIF)                           
```

### 3.1 — Adapter Implementation

```python name=src/adapters/base.py
"""
Base adapter: YAML config → external system format.
Each legacy system gets its own adapter subclass.
"""
from abc import ABC, abstractmethod
from pathlib import Path
from typing import Any, Dict
import yaml

class BaseAdapter(ABC):
    """Loads YAML config and exports to a target format."""

    def __init__(self, config_dir: Path = Path("config/processes")):
        self.configs: Dict[str, Any] = {}
        for f in config_dir.glob("*.yaml"):
            with open(f) as fh:
                self.configs[f.stem] = yaml.safe_load(fh)

    @abstractmethod
    def export(self, process_name: str) -> Any:
        """Transform internal config to external format."""
        ...
```

```python name=src/adapters/erp_adapter.py
"""
Adapter: YAML process config → ERP-compatible flat records.
"""
import csv
import io
from src.adapters.base import BaseAdapter

class ERPAdapter(BaseAdapter):
    """Exports process configs as flat CSV rows for ERP import."""

    def export(self, process_name: str) -> str:
        proc = self.configs[process_name]
        output = io.StringIO()
        writer = csv.writer(output)

        # Header expected by ERP
        writer.writerow([
            "PROCESS_CODE", "MATERIAL_CODE", "SETUP_COST_USD",
            "PER_PART_COST_USD", "TOOLING_COST_USD",
            "MIN_WALL_MM", "TOLERANCE_IT", "MIN_QTY"
        ])

        cost = proc.get("cost", {})
        geo = proc.get("geometry", {})
        batch = proc.get("batch", {})

        # Flatten: one row per material
        for mat in proc.get("materials", []):
            tooling = cost.get("tooling_cost_usd")
            tooling_mid = sum(tooling) / 2 if tooling else 0

            writer.writerow([
                process_name.upper(),
                mat.upper(),
                cost.get("setup_fixed_usd", 0),
                cost.get("per_part_rate_usd", 0),
                tooling_mid,
                geo.get("min_wall_thickness_mm", ""),
                geo.get("tolerance_it_grade", ""),
                batch.get("economic_min_qty", 1),
            ])

        return output.getvalue()
```

```python name=src/adapters/json_api_adapter.py
"""
Adapter: YAML process config → JSON for REST API consumers (MES, PLM, dashboards).
"""
import json
from src.adapters.base import BaseAdapter

class JSONAPIAdapter(BaseAdapter):
    """Exports process configs as JSON, with null handling and type coercion."""

    def export(self, process_name: str) -> str:
        proc = self.configs[process_name]
        # Replace None/null with sensible defaults for consumers
        # that can't handle nulls
        clean = self._sanitize(proc)
        return json.dumps(clean, indent=2)

    def _sanitize(self, obj):
        if isinstance(obj, dict):
            return {k: self._sanitize(v) for k, v in obj.items()}
        if isinstance(obj, list):
            return [self._sanitize(v) for v in obj]
        if obj is None:
            return 0  # or "" — depends on consumer contract
        return obj
```

## 4. When to Use Something Other Than YAML

| Scenario | Better Choice | Why |
|---|---|---|
| **>50 process variants**, multi-user editing | **Lightweight DB** (SQLite, PostgreSQL) with YAML export | Concurrent writes, query speed, audit logging |
| **Real-time process control** (MES integration) | **OPC-UA / MQTT with JSON payloads** | YAML file I/O is too slow and not event-driven |
| **Regulatory / aerospace / medical traceability** | **PLM-native storage** (Teamcenter, Windchill) + YAML as cache | Audit trail must live in the certified system |
| **Supplier data exchange** | **Excel or CSV** (sad but true) | Suppliers won't adopt your YAML schema |
| **CAD integration** | **CAD API objects** (e.g., SolidWorks custom properties) | CAD tools don't read external configs — you push data in via API |
| **Large-scale cost analytics** | **Parquet / columnar database** | YAML is terrible for analytical queries across thousands of parts |

## 5. Hybrid Approach: YAML + Database

For production systems, the most robust pattern is:

```python name=src/config_store.py
"""
Hybrid config store:
- YAML files are the human-editable source of truth (in Git).
- On startup / CI, YAML is validated, flattened, and loaded into a DB.
- All runtime queries go through the DB.
- Adapters export from the DB to legacy formats.
"""
import sqlite3
import yaml
from pathlib import Path
from src.schemas import ProcessConfig  # Pydantic model

DB_PATH = "data/process_config.db"

def init_db():
    conn = sqlite3.connect(DB_PATH)
    conn.execute("""
        CREATE TABLE IF NOT EXISTS process_config (
            process_name TEXT,
            material TEXT,
            attribute TEXT,
            value REAL,
            unit TEXT,
            updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
            PRIMARY KEY (process_name, material, attribute)
        )
    """)
    return conn

def sync_yaml_to_db():
    """Load all YAML configs, validate, and insert into DB."""
    conn = init_db()
    for f in Path("config/processes").glob("*.yaml"):
        with open(f) as fh:
            raw = yaml.safe_load(fh)

        # Validate with Pydantic — fails fast on bad data
        config = ProcessConfig(**raw)

        # Flatten and insert
        for mat in config.materials:
            rows = [
                (f.stem, mat, "setup_fixed_usd", config.cost.setup_fixed_usd, "USD"),
                (f.stem, mat, "min_wall_thickness_mm",
                 config.geometry.min_wall_thickness_mm, "mm"),
                (f.stem, mat, "tolerance_it_grade",
                 config.geometry.tolerance_it_grade, "IT"),
                # ... more attributes
            ]
            conn.executemany(
                "INSERT OR REPLACE INTO process_config VALUES (?,?,?,?,?, CURRENT_TIMESTAMP)",
                rows,
            )
    conn.commit()
    conn.close()
```

### Workflow

```
Engineer edits YAML → Git commit → CI pipeline runs:
   1. Pydantic validation (rejects bad configs)
   2. sync_yaml_to_db() (flattens to SQLite/Postgres)
   3. Adapters export to ERP CSV, MES JSON, etc.
   4. Tests verify round-trip fidelity
```

---

## 6. Concrete Recommendation

| Step | Action |
|---|---|
| **Keep YAML** | As the human-editable, Git-versioned source of truth. It's unbeatable for readability and diffing. |
| **Add Pydantic validation** | So YAML flexibility doesn't become YAML chaos. |
| **Build an adapter layer from day one** | Even if you only need CSV export today, the pattern pays for itself the moment a second system needs data. |
| **Plan for a DB backing store** | Once you exceed ~20 config files or need audit/query capabilities, sync YAML → SQLite/Postgres on CI. |
| **Never let a legacy system read YAML directly** | Always go through a validated, format-specific adapter. |
| **Document the canonical schema** | The biggest integration risk isn't the file format — it's ambiguity in field definitions (e.g., does "cost" include tooling amortization or not?). A schema doc solves this regardless of format. |

**Bottom line:** YAML is the right internal format for your DFM system. The mistake would be assuming legacy systems can consume it directly. Build the adapter layer early, keep it thin, and test it against real legacy imports. The architecture effort is modest; the integration pain it prevents is enormous.
