Yes — **`TopoDS_Compound` is a perfectly fine import result**, but **it is not the best working representation** for either rule engineering or lightweight ML.

Treat it as a **container**, not as your feature-recognition data model.

## Short answer

Your pipeline should be:

1. **STEP reader outputs `TopoDS_Compound`**
2. **Traverse compound into `TopoDS_Solid` / `TopoDS_Shell` / faces / edges**
3. **Normalize each solid into an attributed B-rep graph**
4. **Run rule-based candidate generation**
5. **Run lightweight ML only on candidate regions**

So:

- **Best for import:** `TopoDS_Compound`
- **Best for recognition:** **per-solid attributed face adjacency graph**
- **Best for ML:** **candidate-level feature vectors** or **small region graphs**

That is the practical answer for holes, counterbores/countersinks, slots/pockets, and ribs/fins.

# 1. Is `TopoDS_Compound` the right output?

## What it is good for
A `TopoDS_Compound` is a common top-level shape after STEP import because STEP may contain:

- one part
- multiple disconnected solids
- an assembly-like structure
- reference geometry
- nested compounds

So it is a good **raw import artifact**.

## What it is bad for
For feature extraction, a compound is too coarse because:

- features belong to **individual solids**, not arbitrary compound containers
- adjacency, inside/outside reasoning, and concavity are much easier **within a solid**
- ML examples should usually be built per solid or per candidate region
- compounds may mix actual bodies with junk geometry or multiple parts

So I would say:

> **Keep `TopoDS_Compound` as input, but immediately decompose it into solids and build your own canonical representation.**

# 2. Best working representation for your use case

For your narrow subset, the best internal representation is:

## A. Per-solid attributed face graph
For each `TopoDS_Solid`:

- **node** = face
- **edge** = two faces share a topological edge

Node attributes:
- surface type: plane / cylinder / cone / sphere / torus / bspline
- area
- centroid
- normal or axis
- orientation
- UV bounds
- curvature hints
- analytic parameters:
  - cylinder radius
  - cone semi-angle
  - plane normal
- bounding box
- whether face is likely external/opening-facing

Edge attributes:
- shared edge type: line / circle / ellipse / spline
- edge length
- convex / concave
- dihedral angle
- tangent continuity if useful

This is the **best base representation** for both:
- rule engineering
- lightweight ML

## B. Candidate region objects
Do not run ML over the full solid first.

Instead create candidate objects like:
- cylindrical hole candidate
- coaxial stepped-hole candidate
- depressed prismatic region candidate
- thin-wall protrusion candidate

Each candidate stores:
- participating faces
- local graph neighborhood
- geometric descriptors
- derived measurements

This is where ML should operate.

# 3. Why this is better than mesh-first

You *could* mesh the solid and run ML on triangles or point clouds, but for your target features that is usually worse:

## B-rep advantages
- exact cylinders/cones/planes are preserved
- diameters/depths are precise
- topological adjacency is explicit
- better interpretability
- lower compute
- easier rule writing

## Mesh disadvantages
- lose exact analytic geometry
- harder to detect counterbores/countersinks robustly
- harder to distinguish ribs vs arbitrary thin geometry
- more preprocessing noise

For **holes, stepped holes, slots/pockets, ribs/fins**, B-rep is the right starting point.

# 7. Canonical data model you should create

You should build your own representation instead of carrying `TopoDS_*` everywhere.

Something like:

```cpp name=feature_graph_types.hpp
struct FaceData {
    int id;
    TopoDS_Face face;
    std::string surface_type;   // plane, cylinder, cone, ...
    double area;
    gp_Pnt centroid;
    gp_Dir normal;              // for planar face or sampled normal
    gp_Pnt axis_point;          // if analytic axis exists
    gp_Dir axis_dir;            // cylinder/cone
    double radius;              // cylinder if applicable
    double semi_angle;          // cone if applicable
    Bnd_Box bbox;
};

struct AdjacencyData {
    int face_a;
    int face_b;
    TopoDS_Edge shared_edge;
    std::string edge_type;      // line, circle, ellipse, spline
    double length;
    double dihedral_angle;
    bool concave;
    bool convex;
};

struct SolidGraph {
    TopoDS_Solid solid;
    std::vector<FaceData> faces;
    std::vector<AdjacencyData> adjacencies;
};
```

Then candidate-level structures:

```cpp name=feature_candidates.hpp
struct HoleCandidate {
    int id;
    std::vector<int> face_ids;
    gp_Pnt axis_point;
    gp_Dir axis_dir;
    double diameter;
    double depth;
    bool through;
    bool has_counterbore;
    bool has_countersink;
    std::vector<double> coaxial_radii;
};

struct PocketCandidate {
    int id;
    std::vector<int> face_ids;
    std::vector<int> bottom_faces;
    std::vector<int> side_faces;
    double depth;
    double opening_area;
    bool open_boundary;
};

struct RibCandidate {
    int id;
    std::vector<int> face_ids;
    double thickness;
    double height;
    double length;
    gp_Dir main_direction;
};
```

That is the level where rules and ML become manageable.

# 8. Rule engineering and ML split for each feature family

This is the most important design choice.

## A. Holes
### Rule engineering should do most of the work
Candidate generation:
- find cylindrical faces
- merge tangent/connected coaxial cylinders
- find circular edge loops
- determine if cylinder opens to outside
- determine end condition

ML use:
- classify ambiguous cylinders:
  - hole vs pin boss vs shaft seat vs cosmetic recess
- classify subtype if rule confidence is weak

### Good candidate features for ML
- radius
- axial extent
- number of coaxial faces
- number/type of end faces
- open-on-both-ends
- cone attached near opening
- larger coaxial cylinder attached
- neighbor surface-type histogram
- local concavity pattern

For holes, ML is secondary.

---

## B. Counterbores / countersinks
Again mostly rule-driven.

### Counterbore cues
- two coaxial cylinders
- larger radius near entrance
- shoulder face between diameters

### Countersink cues
- cylinder + cone
- cone near opening
- standard-ish angle often appears, but do not hard-code too aggressively

### ML role
Only resolve noisy edge cases.

---

## C. Slots / pockets
This is where ML starts to help more.

### Rule-based candidate generation
Find **concave depressed regions**:
- planar or near-planar bottom faces
- surrounding side walls
- depth relative to neighboring top/opening faces

Then classify:
- slot
- pocket
- open pocket
- closed pocket
- elongated pocket
- circular pocket

### Useful ML descriptors
- number of bottom faces
- number of side faces
- opening perimeter / area
- depth
- length-to-width ratio
- fraction of curved boundaries
- openness count
- side-face parallelism
- shape regularity of opening loop
- local blend density

A small tree model can be very effective here.

---

## D. Ribs / fins
Strongest case for hybrid rule + ML.

### Rule-based candidate generation
Look for protruding thin-wall structures:
- thin relative to height/length
- connected to larger base
- two large roughly parallel faces
- narrow thickness direction

### Hard part
Many things look like thin walls:
- outer walls
- flanges
- tabs
- bosses with small thickness
- decorative fins
- partition walls

### ML descriptors
- thickness
- height
- length
- aspect ratios
- attachment area to base
- number of free edges
- orientation relative to main part
- local thickness variance
- neighborhood volume ratio
- face normal distribution

Here ML can materially improve classification quality.

# 14. Established practice vs emerging research

For your target features, the practical standard is still:

- graph/topology analysis
- rule-based feature recognition
- optional classifier on handcrafted descriptors

This is scalable and compute-light.

## Emerging research

- learned segmentation on CAD topology
- foundation models for CAD semantics

These are promising, but for your stated constraints, they are usually **Phase 3+, not Phase 1**.
